Why it exists

For the point where manual wiring starts to spread.

Manual wiring is often the best starting point. Injex helps when the same dependency graph appears in app startup, CLI commands, workers, and tests. Keep one composition root while application code stays free from framework-specific dependency hooks.

Services

Wire repositories, gateways, clients, and use cases once at startup.

Workers

Create one scope per job or message while reusing long-lived clients.

Tests

Override external dependencies inside one explicit `with` block.

CLIs

Share settings, API clients, and commands without module singletons.

Quick start

Normal Python types. Small container API.

Register services at the boundary, validate the graph, then resolve application services from one place. 

from injex import Container

class Repository:
    pass

class Service:
    def __init__(self, repo: Repository):
        self.repo = repo

container = Container()
container.add_singleton(Repository)
container.add_transient(Service)
container.assert_valid()

service = container.resolve(Service)

Positioning

Small by design.

Injex is not trying to be a configuration framework. It focuses on the boring 80%: typed constructor injection, scoped lifetimes, factories, named registrations, test overrides, and validation without object creation.
